ENCRYPTED CREDENTIAL STORE (structure, no values)
-------------------------------------------------
- Product credential bundle (V2) is written through the encrypted CredentialStore
  (AES-256-GCM, AAD-bound) via the atomic temp→fsync→rename writer; class/id are
  SecretClass::ProviderOAuthRefresh + the provider id.
- ProductActivationV1 is written under its own SecretClass::ProductActivation, id
  "activation:<provider>", carrying only {provider_id, credential_generation (UUIDv4),
  oauth_policy_fingerprint (SHA-256 of the official OAuth tuple), harness_contract_version}.
- OnboardingAttemptJournalV1 is written under SecretClass::OnboardingAttemptJournal, id =
  a domain-separated SHA-256 of the attempt id / generation (never the raw id); loaded with
  a bounded no-follow read (get_bounded, ≤65536 plaintext / ≤98304 envelope bytes).
- Readiness authority = provider_ready(p): a valid Active V2 bundle AND an activation
  matching {generation + official policy fingerprint + harness contract version} AND a
  passing static harness attestation. The TTL'd journal is NEVER the authority.

NEGATIVE LOCAL-CLI (experimental never satisfies product readiness)
-------------------------------------------------------------------
- An experimental local-CLI credential resolves ONLY under agent-subscription-experimental,
  ONLY on an Absent product state, via try_experimental_only_provider — it has no bundle and
  no activation, so provider_ready is false and status connected/claude are false; the walled
  experimental turn path stays available but never sets product readiness/activation.
- Product builds never read ~/.claude / ~/.codex.
Tests: experimental_credential_never_sets_product_readiness,
       product_oauth_provider_does_not_read_desktop_cli_fallbacks,
       product_build_does_not_read_dot_claude, product_build_does_not_read_dot_codex,
       corrupt_product_credential_requires_reconnect_without_local_fallback.

OFFICIAL-OAUTH OVERRIDE REJECTION
---------------------------------
- A non-official OAuth recipe (endpoint/client/scope override) is rejected by load_oauth_config
  ("OAuth recipe does not match the official product policy").
- An activation minted under a non-official policy fingerprint never matches → provider_ready false.
Tests: product_oauth_config_rejects_endpoint_client_scope_override,
       product_oauth_config_accepts_only_compiled_official_recipe,
       product_oauth_override_cannot_satisfy_official_handoff.
